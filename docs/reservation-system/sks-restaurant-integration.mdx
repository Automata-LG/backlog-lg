### 2.2 3계층 구조 (Host → Space → Unit)

<Card title="3계층 구조" icon="layer-group">
  - **Host**: 최상위 시설 (다이닝, 카페, 아트센터)
  - **Space**: 중간 구역 (홀, 룸, 테라스, 좌석구역)
  - **Unit**: 최소 예약 단위 (테이블, 좌석)
</Card>

#### ReservationUnit (신규)
```prisma
model ReservationUnit {
  id          String @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  sn          String @unique @db.VarChar(50)
  name        String @db.VarChar(35)
  description String
  minCapacity Int    @default(0)
  maxCapacity Int    @default(0)
  metadata    Json?

  space        ReservationSpace @relation(fields: [spaceId], references: [id])
  spaceId      String           @db.Uuid
  reservations Reservation[]

  index     String
  isActive  Boolean  @default(true)
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@unique([spaceId, index])
  @@map(name: "reservationUnit")
}
```

## 3. 레스토랑 특화 운영 정책

### 3.1 등급별 차등 오픈

<Card title="등급별 예약 가능 기간" icon="calendar">
  - **다이아몬드 블랙 (AS00)**: 조회일 기준 익익월까지 예약 가능
  - **나머지 등급 (AS01~AS05)**: 조회일 기준 익월까지 예약 가능
  - 매월 1일 오전 10시 신규 타임슬롯 오픈
</Card>

```typescript

  openStartAtByGrade: {

    'AS01': '2025-02-01T10:00:00',  // 다이아몬드: 2월 1일부터 예약 가능

    'AS03': '2025-02-01T10:00:00',  // 골드: 2월 1일부터 예약 가능
    'AS04': '2025-02-01T10:00:00',  // 실버: 2월 1일부터 예약 가능

  },
  modificationByDays: 2,  // 수정/취소 마감: D-2일 23:59:59
```

### 3.2 수정/취소 정책

<Note>
  **역할별 수정/취소 데드라인**

  - **일반 사용자**: 이용 N일 전 23:59:59까지 (modificationByDays 설정값)
    - 기본값: 2일 전 23:59:59
    - 예: 3월 15일 예약 → 3월 13일 23:59:59까지 수정/취소 가능

  - **관리자**: 이용 시작 시간 직전까지
    - 코드 레벨에서 처리 (별도 설정 없음)
    - 예: 3월 15일 12:00 예약 → 3월 15일 11:59까지 수정/취소 가능
</Note>

#### 수정/취소 변경 추적

### 3.3 타임슬롯 운영 방식

| 구분 | 아트센터 | SKS 레스토랑 |
|------|----------|--------------|
| **운영 단위** | 공연 시간별 슬롯 | 일 단위 통합 슬롯 |
| **시간 선택** | 정해진 공연 시간 | 30분 단위 방문 시간 |
| **정원 관리** | 슬롯별 좌석 수 | 일별 테이블 수 |
| **예약 단위** | 좌석 개수 | 테이블 + 인원 |
| **오픈 방식** | BATCH/SEQUENTIAL | GRADED (등급별 차등) |
| **예약 마감** | modificationByHours | reservationByDays |
| **수정/취소 마감** | modificationByHours | modificationByDays |

```typescript
// 레스토랑은 일 단위로 하나의 타임슬롯 생성
const restaurantTimeSlot: ReservationTimeSlot = {
  id: 'uuid',
  contentId: '2025-09-01-dining-content-id',
  startAt: new Date('2025-09-01 11:00:00'),
  endAt: new Date('2025-09-01 21:00:00'),    // 당일 운영 종료 시간
  totalCapacity: 50,                          // 전체 수용 인원
  minGroupSize: 2,                            // 최소 예약 인원
  maxGroupSize: 8,                            // 최대 예약 인원
  openConfig: {                               // GRADED 타입 설정
    openType: 'GRADED',
    openStartAtByGrade: {
      'AS00': '2025-01-01T10:00:00',         // 2개월 전 오픈
      'AS01': '2025-02-01T10:00:00'          // 1개월 전 오픈
    },
    reservationByDays: 1,                    // D-1일 23:59:59까지 예약 가능
    modificationByDays: 2                    // D-2일 23:59:59까지 수정/취소 가능
  },
  isActive: true
};
```

### 3.4 물리적 제약 자동 처리

<Note>
  **물리적 제약은 Space/Unit으로 자동 처리**

  레스토랑의 5-6인 일일 2팀 제한은 별도 정책이 아닙니다:
  - 6인 테이블이 물리적으로 2개만 존재
  - 5-6인 그룹은 6인 테이블에만 배정 가능
  - 따라서 자동으로 일일 최대 2팀 제한
</Note>

### 4.2 빈자리 알림 (WaitingList)

<Note>
  **빈자리 알림 기능**
  - 예약이 만석인 날짜에 대한 대기 등록
  - 취소 발생 시 대기자에게 순차적 알림
  - 선착순 예약 재오픈
</Note>

#### 대기 등록 구조
```typescript
interface IWaitingListEntry {
  id: string;
  memberId: string;
  memberGrade: TMembershipGrade;
  contentId: string;
  timeSlotId: string;          // 특정 날짜의 타임슬롯
  preferredVisitTime?: string;  // 희망 방문 시간 (HH:MM)
  preferredSpaceType?: 'ROOM' | 'HALL';  // 룸/홀 선호
  groupSize: number;
  registeredAt: Date;
  notifiedAt?: Date;           // 알림 발송 시각
  status: 'WAITING' | 'NOTIFIED' | 'EXPIRED' | 'CONFIRMED';
}
```

#### 빈자리 알림 플로우
```typescript
// 예약 취소 시 대기자 알림
async function notifyWaitingList(cancelledReservation: IReservation) {
  // 취소된 예약의 타임슬롯 대기자 조회
  const waitingList = await getWaitingList({
    timeSlotId: cancelledReservation.timeSlotId,
    status: 'WAITING',
    groupSize: { lte: cancelledReservation.groupSize }
  });
  
  // 등급별 우선순위 + 등록 시간 순 정렬
  const sortedList = waitingList.sort((a, b) => {
    const gradeOrder = { AS00: 0, AS01: 1, AS02: 2, AS03: 3, AS04: 4, AS05: 5 };
    if (gradeOrder[a.memberGrade] !== gradeOrder[b.memberGrade]) {
      return gradeOrder[a.memberGrade] - gradeOrder[b.memberGrade];
    }
    return a.registeredAt.getTime() - b.registeredAt.getTime();
  });
  
  // 상위 N명에게 알림 (보통 3-5명)
  const notifyCount = Math.min(5, sortedList.length);
  for (let i = 0; i < notifyCount; i++) {
    await sendVacancyNotification(sortedList[i]);
    await updateWaitingStatus(sortedList[i].id, 'NOTIFIED');
  }
  
  // 24시간 후 만료 처리 스케줄링
  scheduleExpiration(sortedList.slice(0, notifyCount), 24 * 60 * 60 * 1000);
}
```

## 5. 예약 상태 관리 (FSM)

### 5.1 기존 FSM 활용

<Card title="기존 FSM 그대로 활용" icon="diagram-project">
  SKS 레스토랑의 모든 상태는 기존 ReservationStatus enum으로 완벽하게 표현 가능합니다.
</Card>

```typescript
// 기존 ReservationStatus enum 그대로 활용
enum ReservationStatus {
  PENDING = 'PENDING',
  CONFIRMED = 'CONFIRMED',
  CANCELLED = 'CANCELLED',
  USED = 'USED',
  NOSHOW = 'NOSHOW'
}
```

### 5.2 레스토랑 관점 매핑

| SKS 상태 | 기존 ReservationStatus | 구분 방법 |
|----------|------------------------|-----------|
| 예약대기 | PENDING | status = 'PENDING' |
| 예약확정 | CONFIRMED | status = 'CONFIRMED' & unitId = null |
| 예약확정_좌석배정 | CONFIRMED | status = 'CONFIRMED' & unitId != null |
| 예약취소_유저 | CANCELLED | reasonCode = 'USER_CANCEL' |
| 예약취소_관리자 | CANCELLED | reasonCode = 'ADMIN_CANCEL' |
| 이용완료 | USED | status = 'USED' |
| 노쇼 | NOSHOW | status = 'NOSHOW' |

#### 상태 전환 규칙
```typescript
// 레스토랑 예약 생성 (즉시 확정)
USER_CREATE_CONFIRMED → CONFIRMED

// 좌석 배정 (상태 변경 없음, 필드만 업데이트)
CONFIRMED + updateUnitId

// 체크인 처리
ADMIN_MARK_USED → USED

// 취소 (D-2까지)
USER_CANCEL → CANCELLED
```

### Space/Unit 기반 가용성 검증
```typescript
async function validateResourceAvailability(
  reservationRequest: any,
  spaceId: string
) {
  // 1. 해당 공간의 Unit 조회
  const space = await db.reservationSpace.findUnique({
    where: { id: spaceId },
    include: {
      units: {
        where: { isActive: true }
      }
    }
  });

  // 2. 요청 인원에 맞는 Unit 찾기
  const availableUnit = space?.units
    .find(unit => {
      return reservationRequest.guests >= unit.minCapacity && 
             reservationRequest.guests <= unit.maxCapacity;
    });

  if (!availableUnit) {
    throw new Error('적합한 테이블/좌석이 없습니다');
  }

  // 3. 해당 Unit의 예약 상태 확인
  const existingReservation = await db.reservation.findFirst({
    where: {
      unitId: availableUnit.id,
      date: reservationRequest.date,
      status: { in: ['PENDING', 'CONFIRMED'] }
    }
  });

  if (existingReservation) {
    throw new Error('이미 예약된 테이블/좌석입니다');
  }

  return availableUnit;
}
```

## 9. 에러 처리

<Warning>
  **예약 시스템 에러 처리**

  1. **중복 예약 방지**
     - 동일 날짜에 이미 해당 컨텐츠에 대한 예약 존재 시
     - 에러 메시지: "이미 해당 날짜에 예약이 존재합니다"

  2. **예약 미 오픈**
     - 블랙: D+60일 초과
     - 일반: D+30일 초과
     - 에러 메시지: "예약이 아직 오픈 되지 않은 일자 입니다"

  3. **테이블 배정 오류**
     - 5-6인 그룹 일일 2팀 제한 초과
     - 해당 테이블 모두 예약 완료

  4. **잔여좌석 없음**
     - 예약 완료 직전 다른 사용자가 먼저 예약
     - 에러 메시지: "예약가능한 좌석이 없습니다"

  5. **예약/취소/변경 시간 경과**
     - 수정/취소 가능 시간 경과
     - 에러 메시지: "예약/취소/변경 가능한 시간이 아닙니다"
</Warning>
